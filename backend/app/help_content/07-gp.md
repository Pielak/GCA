# Área de Gestão de Projeto (GP)

Entrada: `/projects` → `/projects/:id` — acesso para membros ativos do projeto.

O GP é **soberano do projeto**: tem acesso a todas as funcionalidades (OCG, ingestão, Gatekeeper, Arguidor, CodeGen, testes, doc, releases). Dev, Tester e QA compartilham o projeto com papéis de dia-a-dia, mas a permissão plena está com o GP.

## Meus Projetos (`/projects`)

- **Lista** de projetos onde você é membro ativo, com status, seu papel e progresso.
- Indicador de pendências por projeto (items do Arguidor esperando resposta, quarentenas pendentes, etc).
- Clique em um projeto para ir ao dashboard dele.

## Dashboard do Projeto (`/projects/:id`)

Visão consolidada:

- **Scores OCG** em radar chart dos 7 pilares (P1–P7).
- **Detalhamento** por pilar com nível de aderência.
- **Status** do projeto: READY, NEEDS_REVIEW, AT_RISK ou BLOCKED + versão atual do OCG.
- **Stack recomendada** resumida por camada.
- **Billing de IA do projeto**: custo total em USD, tokens, número de chamadas + breakdown por operação e provedor.
- **Equipe** com papel de cada membro.
- Atalhos para todas as abas do pipeline.

## Configurações (`/projects/:id/settings`)

Abas internas:

### Questionário Técnico
- Questionário submetido inicialmente + análise + correções enviadas pelo GP depois do Arguidor.
- PDF alternativo para preenchimento offline.

### Repositório Git
- Conecta GitHub, GitLab ou Bitbucket.
- Token de acesso pessoal (PAT) criptografado.
- Define branch default.
- **Um mesmo repositório não pode ser compartilhado entre projetos** — o sistema bloqueia e mostra qual projeto já usa aquele repo.

### Provedor de IA do projeto
- **Adicionar/remover** provedor: Anthropic, OpenAI, DeepSeek, Ollama.
- **Validar a chave** antes de salvar.
- **Definir o padrão** do projeto.
- Essas chaves são **separadas** das globais da instância — custo da IA do dia-a-dia fica com o projeto.

### Equipe
- Convidar Dev, Tester, QA ou outro GP por email.
- **Convidar outro GP** do mesmo projeto (co-gestão).
- **Transferir a soberania** para outro membro ativo.
- Revogar convite pendente.
- Remover membro (com proteção de último GP).
- **Listagem separa "Convites Pendentes" de "Membros da Equipe"** — um usuário só aparece em Membros quando efetivamente aceita o convite (estado canônico `joined_at IS NOT NULL`). Convites emitidos mas não aceitos ficam só na seção Convites Pendentes.

### Adicionar Integrações do projeto
- Painéis **Issue Tracker** (Jira/Trello) e **Notificações** (Slack/Teams) disponíveis na aba Integrações.
- Credenciais encrypted no vault (pgcrypto) — nunca trafegam em plaintext.
- Eventos canônicos do pipeline disparam notificação conforme opt-in.

### Repos Externos
- Atalho para `/projects/:id/external-repos` (ver abaixo).

### Integrações (MVP 20)

Aba onde o GP conecta as 3 categorias de integração externa.

![Painel de Provedores de IA — mesmo padrão visual usado pela aba Integrações](/images/help/provedores.png)

**Issue Tracker** (Jira ou Trello)
- Providers suportados V1: `jira`, `trello`.
- Credenciais por projeto (encrypted no vault via `pgcrypto`): email + api_token do Atlassian (Jira); api_key + user token (Trello); webhook_secret opcional em ambos.
- **Status mapping configurável** — GP pode mapear status específicos do seu fluxo (ex: "Em análise pelo jurídico") para os 5 canônicos do GCA: `todo`, `in_progress`, `review`, `done`, `cancelled`.
- Webhook bidirecional: módulo aprovado no GCA → cria issue no tracker; status muda no tracker → atualiza `ExternalIssue` local. HMAC + replay prevention obrigatórios.
- Integrações sob demanda (Linear, Asana, GitHub Issues, Monday, ClickUp) custam ~1.5-2d de desenvolvimento cada — pedir ao Admin se precisar.

**Security Scanners** (Sonar, Snyk, gitleaks)
- Consome findings do scanner que sua empresa já paga — GCA não reimplementa SAST.
- P7 do OCG passa a ser recalculado determinísticamente (fórmula no [cap. 5](?section=05-ocg)).
- GP pode marcar finding como `accepted_risk` com justificativa mínima de 10 caracteres; fica na trilha de auditoria SHA-256.

**Notifier** (Slack)
- Incoming Webhook URL por projeto.
- 6 eventos canônicos opt-in: `MODULE_APPROVED`, `OCG_CONSOLIDATED`, `CODEGEN_COMPLETED`, `ERS_REGENERATED`, `SECURITY_FINDING_HIGH`, `BACKUP_FAILED`.
- Lista de opt-in vazia = opted in em todos (default).
- **Modo link-only** disponível — mensagem vai ao canal com link pro GCA mas zero payload sensível no corpo (obrigatório em cliente BACEN, ANS, órgão público).
- Falha na notificação NUNCA bloqueia evento no pipeline — é best-effort.

Detalhes operacionais, credenciais passo-a-passo e troubleshooting em [cap. 11 — Integrações Externas](?section=11-integracoes).

## OCG (`/projects/:id/ocg`)

Página central do projeto. Mostra as 12 seções do OCG renderizadas, a versão atual e o `context_health`.

O que você pode fazer aqui:

- **Ver o OCG atual** com detalhamento por seção.
- **Linha do tempo de versões** — histórico de mudanças com origem de cada delta.
- **Ver snapshot** de qualquer versão anterior (somente leitura).
- **Rollback** — voltar para uma versão anterior (cria nova versão, não apaga histórico).
- **Consolidar OCG** — recalcular score composto e status a partir dos pilares. Idempotente.
- **Regenerar OCG** — descartar histórico e refazer do zero (ação destrutiva, com confirmação dupla).
- **Modelo de dados** — tabela com tabelas, índices e FKs derivados do OCG.

Detalhes em [cap. 5 — OCG](?section=05-ocg).

## Repositórios Externos (`/projects/:id/external-repos`)

Vincula repositórios legados ao projeto para o GCA analisar:

- **Adicionar** repo com URL + credenciais.
- O sistema analisa automaticamente: stack detectado, vulnerabilidades, compatibilidade com o GCA.
- **3 abas** no resultado:
  - **Stack Detectado**: linguagem, frameworks, arquivos totais, Docker, CI/CD, testes.
  - **Segurança**: nível de risco (baixo/médio/alto) + lista de vulnerabilidades com versão recomendada.
  - **Compatibilidade**: status geral (compatível / requer adaptação / incompatível), esforço estimado, breakdown por área.
- **Roadmap de adaptação** passo-a-passo.
- A análise do legado alimenta o Arguidor com perguntas específicas.

## Ingestão (`/projects/:id/ingestion`)

- **Drop zone** para upload (PDF, DOCX, XLSX, PNG, JPG, MD — até 50 MB).
- **Vinculação opcional** a um módulo do Roadmap (útil para anexar documentação específica).
- Pipeline assíncrono com **barra de progresso** em tempo real: `queued → extracting_text → analyzing → updating_ocg → regenerating_backlog → completed`.
- **Extração rica** por tipo (DOCX com tabelas, PDF com AcroForm + OCR, XLSX preservando estrutura).
- **Quarentena PII** — documentos com CPF/CNPJ/cartão/telefone ficam retidos até o GP decidir: liberar ou descartar.
- **Detalhe do documento** — análise do Arguidor + gaps + show_stoppers + poor_definitions + improvement_suggestions + módulos candidatos.
- **Deletar** documento com cascata no OCG.

## Gatekeeper (`/projects/:id/gatekeeper`)

![Gatekeeper com score global + radar dos 7 pilares e status por pilar](/images/help/gatekeeper.png)

- **7 pilares** com score + status (ok, warning, blocker).
- **Items rastreados**: gaps, show_stoppers, poor_definitions, improvement_suggestions.
- **Módulos candidatos** derivados das ingestões.
- **Summary** + indicador de bloqueadores ativos.
- **Health do OCG** (confidence, quality).

## Arguidor (`/projects/:id/arguider`)

- **Lista de perguntas dirigidas** geradas a partir das ingestões e da análise de repos legados.
- **Responder** com texto + evidência opcional.
- **Ignorar com motivo** se a pergunta não se aplica ao seu projeto.
- **Resolver** alimenta o OCG de volta (expand ou update).
- **Contexto OCG visível** (score, status, versão, confidence) para você decidir com contexto.

## CodeGen (`/projects/:id/codegen`)

Interface tipo IDE com árvore de arquivos do repositório à esquerda (sidebar colapsável).

![CodeGen com editor inline, árvore de arquivos e botões Preview/Regenerar/Revisar e Salvar](/images/help/codegen.png)

- **Gerar scaffold** a partir do `STACK_RECOMMENDATION` do OCG.
- **8 linguagens canônicas** (9 scaffolders) + Python via LLM.
- **DDL automático** — schema.sql, seed.sql e migrations injetados a partir do modelo de dados.
- **Preview antes do commit** — diff completo.
- **Apply** — cria commit no repositório Git do projeto com mensagem padrão.
- **Regenerar arquivo** específico.
- **Docstrings obrigatórias** em todo código gerado.
- **Validação pós-geração**: pyflakes, esprima, ast.parse, cmake conforme a linguagem.

Detalhes em [cap. 8 — Codegen](?section=08-codegen).

## Documentação Viva (`/projects/:id/docs`)

Organizada em 4 áreas funcionais (todas lado a lado na mesma página):

![Painel de Documentação Viva com 20 docs geradas — Arquitetura via premium, módulos via Ollama local](/images/help/docs_vivas.png)

### LiveDocs (docs geradas por LLM)

- Por módulo: via IA local (Ollama com `qwen2.5-coder:7b` no dogfood; cliente troca pelo modelo que preferir).
- Consolidação geral: via IA premium (Claude, GPT, Gemini — o que o GP configurar).
- **Regenerar** granular por tipo (module_doc, index, architecture).
- **Banner "Stale"** avisa quando o OCG mudou e a doc precisa regenerar.
- **Viewer somente-leitura** dos documentos originais ingeridos.

### ERS Vivo — IEEE 830 (MVP 19)

- Botão **"Regenerar ERS"** produz `docs/ERS.md` e commita no repositório Git do projeto via `git_service` (PAT configurado em `/settings` → Repositório).
- Estrutura canônica IEEE 830-1998 em 4 seções:
  1. Introdução (propósito, escopo, **definições** — alimentado pelo glossário aprovado).
  2. Descrição Geral (perspectiva, funções, usuários, restrições).
  3. Requisitos Específicos — RF / RNF / BR categorizados pelo GP.
  4. Matriz de Rastreabilidade — requisitos × testes × arquivos gerados.
- Mensagem canônica de commit: `docs(ers): regen a partir do OCG v{N} — {razões-stale}`.
- Histórico nativo via `git log -p docs/ERS.md` (decisão binária: zero snapshot em banco).
- **Pré-requisito**: repositório Git conectado. Sem repo o botão fica desabilitado com mensagem explicativa.
- **Freshness**: card mostra quando o ERS ficou stale (OCG mudou, ingestão nova, etc). Regeneração é **manual** — sistema detecta e sinaliza, mas não commita sozinho (evita commits ruidosos no Git do cliente).

### Glossário do projeto (MVP 19.3)

- Extração automática de termos candidatos a partir do corpus já persistido: `module_candidates` (nome + descrição), `arguider_responses`, `gatekeeper_items`, `OCG.PROJECT_PROFILE`.
- 4 heurísticas regex determinísticas (sem LLM no caminho crítico):
  - Siglas soltas (ex: "ANS", "B2B", "V2")
  - "Expansão (SIGLA)" (ex: "Receita Bruta (RB)")
  - "SIGLA: definição" ou "SIGLA — definição"
  - "X é Y" / "X significa Y"
- Idempotência via `UNIQUE (project_id, LOWER(term))` — rodar extração 2x não duplica.
- Ciclo canônico: `candidate` → **GP aprova** → `approved` (entra na seção 1.3 do ERS) OU **GP rejeita** → `rejected` (não volta a aparecer).
- GP pode **editar definição** antes de aprovar ou criar termo manual.
- Trilha de auditoria SHA-256 em cada aprovação/rejeição/edição.

### Matriz de Rastreabilidade (MVP 19.4)

- Query determinística sob demanda (sem view materializada — decisão binária #4 do MVP 19).
- Correlaciona `ModuleCandidate` × `TestSpec.module_id` × `GeneratedModule.module_candidate_id` via LEFT JOIN semântico em Python — requisito aparece mesmo sem spec nem código gerado.
- Ordem canônica IEEE 830: RF → RNF → BR → não-classificados. Numeração 1-based dentro de cada categoria.
- Sumário agregado: total, por categoria, com spec, com código, rastreamento completo (spec + código).
- Mesmo conteúdo é incluído na Seção 4 do ERS ao regenerar.

## Auditoria do Pipeline (`/projects/:id/audit`)

- Lista das fases do pipeline que rodaram no projeto (ingestão, Gatekeeper, Arguidor, CodeGen, QA) com duração, status e contexto.
- Filtro por fase e janela temporal.
- Drill-down por evento.

## Backups do Projeto (`/projects/:id/backups`)

![Lista de backups do projeto — automáticos com hash SHA e botão de rollback](/images/help/backup.png)

- **Backup automático diário** — cron configurável (default `00 12 * * *` America/Sao_Paulo). Lista mostra os últimos 10 snapshots com hash SHA + autor + tipo (Automático/Manual).
- **Catch-up no startup** — se o servidor estava down no horário programado, o backup é executado assim que o backend sobe (evita janela silenciosa).
- **Backup manual** pelo GP ou Admin via botão "Backup agora".
- **Rollback** a partir de qualquer snapshot selecionado (botão "Rollback" na linha).
- **Compartimentalização §2.2** — backups de um projeto nunca aparecem em outro. Snapshot cobre OCG (todas as versões), módulos, test_specs, LiveDocs, external_issues, security_findings, glossário, configurações.
- Download em JSON pra auditoria externa.

## Incidentes (`/projects/:id/incidents`)

- Lista de tickets do projeto.
- **Criar ticket** com severidade (BAIXO, MÉDIO, ALTO, CRÍTICO) + descrição + anexos (até 5 arquivos, 10 MB cada, 9 extensões aceitas).
- Detalhe com comentários + resolução.
- **Roteamento automático por papel** — bugs técnicos para Dev, mudança de escopo para GP, sustentação para Support.
- Contexto obrigatório: `section_reference` (autopreenchido conforme a aba de origem) e `flow_description` (bloqueante — você precisa descrever o fluxo que disparou o problema).

## Métricas do Projeto (`/projects/:id/metrics`)

Controle de custo de IA visível, compartimentalizado por projeto. Não precisa consolidar nada — tudo o que aparece aqui foi consumido por este projeto especificamente.

![Métricas do projeto — breakdown por provider × operation com custo em USD](/images/help/metricas.png)

### Cabeçalho agregado

- **Chamadas LLM** totais no período.
- **Tokens In** / **Tokens Out** acumulados.
- **Custo estimado** em USD (calculado no momento da chamada usando as tabelas de preço do provider).
- Janela ajustável: 24h · 7 dias · 30 dias.

### Tabela provider × operação

Breakdown por combinação única de `provider` (anthropic, openai, ollama, etc) × `operation` (analyzer, pillar_p1..p7, consolidator, codegen_scaffold, livedocs_module, ers_generator, etc) mostrando chamadas + tokens + custo. GP consegue identificar rapidamente qual operação está dominando o custo e decidir se vale trocar pro modelo local pra aquela operação.

### Eventos de auditoria

Lista dos eventos canônicos que disparam neste projeto (EXTERNAL_ISSUE_CREATED, BACKLOG_REGENERATED, CODEGEN_COMPLETED, LIVEDOCS_UPDATED com doc_type=ers, etc) com count no período. Útil pra detectar picos anormais de atividade.

### Autorização

Admin, Support ou membro aceito do projeto. Métricas não cruzam projetos (§2.2).
