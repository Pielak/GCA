  GCA — Perguntas feitas na abertura (solicitação) de um novo projeto
  Fluxo: /solicitar-projeto  (wizard de 2 passos, público, sem login)
  Fontes:
    - frontend/src/pages/SolicitarProjetoPage.tsx
    - frontend/src/data/projectRequestQuestions.ts

Observação: o wizard de solicitação é curto de propósito. As respostas viram
o SEED INICIAL do OCG após o admin aprovar. O questionário completo (49
perguntas) é apresentado DEPOIS, já dentro do projeto — não faz parte da
abertura/solicitação.


───────────────────────────────────────────────────────────────────────────
PASSO 1 — DADOS BÁSICOS (sempre exibidos)
───────────────────────────────────────────────────────────────────────────

1. Seu nome *
   (texto livre, 2–255 caracteres)
   Ex: Maria Silva

2. Seu email *
   (email válido; usado para comunicação; se não tiver conta, será criada
    após aprovação)
   Ex: voce@empresa.com

3. Nome do projeto *
   (texto livre, 3–255 caracteres)
   Ex: Webapp de previsão do tempo

4. Tipo de entregável *
   (seleção única, opções):
     - Novo sistema
     - Aplicativo mobile
     - Módulo / extensão
     - Melhoria em sistema existente
     - Integração
     - Modernização / refatoração
     - ETL / pipeline de dados
     - Sustentação evolutiva
     - Outro (descrever)

4a. Descreva o tipo *  (SÓ se o tipo = "Outro")
    (texto livre, 2–100 caracteres — o Arguidor usa para escolher a stack)
    Ex: Browser Extension, Web Application, CLI Tool…

5. Descrição breve *
   (textarea obrigatória, mínimo 30 e máximo 2000 caracteres)
   "Conte o objetivo, escopo principal, quem vai usar e o problema resolvido."


───────────────────────────────────────────────────────────────────────────
PASSO 2 — REQUISITOS ESPECÍFICOS POR TIPO + PERGUNTAS COMUNS
───────────────────────────────────────────────────────────────────────────

As perguntas a seguir variam conforme o "Tipo de entregável" escolhido no
passo 1. As DUAS PERGUNTAS COMUNS aparecem SEMPRE ao final do passo 2,
independente do tipo.


▌PERGUNTAS COMUNS (sempre apresentadas — qualquer tipo)

C1. Quem são os stakeholders / patrocinadores principais? *
    (textarea, mínimo 10 caracteres)
    Ex: Diretor de TI (Maria), Gerência Comercial (João)

C2. Como vocês saberão que o projeto foi um sucesso? *
    (textarea, mínimo 20 caracteres; listar 1–3 critérios objetivos)
    Ex: Reduzir tempo de cadastro em 50%; suportar 1k usuários simultâneos.


▌TIPO 1 — Novo sistema  (new_system)

N1. Quem usará o sistema? (público-alvo) *
    (texto livre)
    Ex: Vendedores externos da empresa

N2. Volume esperado de usuários simultâneos *
    (seleção):
      - até 50
      - 50–500
      - 500–5.000
      - 5.000+

N3. Necessita autenticação / login? *
    (seleção):
      - Não
      - Sim — usuário e senha
      - Sim — SSO corporativo (SAML/OIDC)

N4. Onde será hospedado? *
    (seleção):
      - Cloud pública (AWS/Azure/GCP)
      - On-premise (servidor próprio)
      - Ainda não decidido


▌TIPO 2 — Aplicativo mobile  (mobile_app)

M1. Plataforma alvo *
    (seleção):
      - Android apenas
      - iOS apenas
      - Ambos (cross-platform)

M2. Precisa funcionar offline? *
    (seleção):
      - Não
      - Parcialmente (alguns recursos)
      - Sim, totalmente

M3. Quais recursos do dispositivo são usados? *
    (textarea, mínimo 10 caracteres)
    Help: Câmera, GPS, push notifications, biometria, NFC, etc.
    Ex: GPS para rota e câmera para QR code.

M4. Backend / API *
    (seleção):
      - Já existe — vamos integrar
      - Será construído junto
      - Ainda não decidido


▌TIPO 3 — Módulo / extensão  (module)

Mo1. Sistema hospedeiro / ecossistema *
     (texto livre)
     Help: Onde o módulo vai rodar? (ex: SAP, Salesforce, Chrome, WordPress)
     Ex: Extensão para Google Chrome

Mo2. Como será distribuído / instalado? *
     (textarea, mínimo 10 caracteres)
     Ex: Publicar na Chrome Web Store; instalar via marketplace.

Mo3. Que APIs ou pontos de extensão do hospedeiro serão usados? *
     (textarea, mínimo 10 caracteres)
     Ex: Chrome Storage API, content_scripts em páginas X.


▌TIPO 4 — Melhoria em sistema existente  (enhancement)

E1. Sistema atual a ser melhorado *
    (textarea, mínimo 20 caracteres)
    Help: Nome, propósito e escala (quantos usuários, há quanto tempo no ar).
    Ex: Portal de RH interno, ~300 usuários, 4 anos no ar.

E2. Stack tecnológica atual *
    (texto livre)
    Help: Linguagens, frameworks, banco de dados.
    Ex: PHP 7.4 + MySQL + jQuery

E3. O que precisa ser melhorado ou adicionado? *
    (textarea, mínimo 30 caracteres)
    Ex: Adicionar tela de relatórios; performance da tela X; nova integração.


▌TIPO 5 — Integração  (integration)

I1. Sistema(s) de origem *
    (textarea, mínimo 10 caracteres)
    Ex: ERP TOTVS Protheus (versão 12)

I2. Sistema(s) de destino *
    (textarea, mínimo 10 caracteres)
    Ex: CRM Salesforce + Data Warehouse interno

I3. Modo de sincronização *
    (seleção):
      - Tempo-real (event-driven)
      - Quase-real (polling alguns minutos)
      - Batch diário
      - Batch sob demanda

I4. Volume estimado por execução *
    (seleção):
      - Baixo (<1k registros)
      - Médio (1k–100k)
      - Alto (100k–10M)
      - Muito alto (>10M)


▌TIPO 6 — Modernização / refatoração  (modernization)

Mod1. Stack legada (a ser substituída) *
      (texto livre)
      Ex: COBOL + DB2 em mainframe

Mod2. Stack alvo desejada (se já decidida) *
      (texto livre)
      Help: Se ainda não decidiu, escreva "a definir".
      Ex: Java 21 + PostgreSQL em containers

Mod3. Estratégia de migração preferida *
      (seleção):
        - Big-bang (substituição completa)
        - Strangler fig (incremental por módulo)
        - Lift-and-shift (só infra)
        - A definir

Mod4. Tolerância a downtime *
      (seleção):
        - Zero (24/7)
        - Pequena janela noturna
        - Fim de semana
        - Sem restrição


▌TIPO 7 — ETL / pipeline de dados  (etl)

T1. Fontes de dados *
    (textarea, mínimo 10 caracteres)
    Ex: API X, banco Oracle Y, planilhas em S3.

T2. Destinos de dados *
    (textarea, mínimo 10 caracteres)
    Ex: Data warehouse Snowflake; relatórios em Power BI.

T3. Periodicidade *
    (seleção):
      - Tempo-real (streaming)
      - Horário
      - Diário
      - Semanal
      - Sob demanda

T4. Há regras de qualidade / validação obrigatórias? *
    (textarea, mínimo 10 caracteres)
    Ex: Sem nulos em campo X; deduplicação por CPF; conversão BRL→USD.


▌TIPO 8 — Sustentação evolutiva  (maintenance)

Su1. Sistema a ser mantido *
     (textarea, mínimo 20 caracteres)
     Ex: Portal interno de pedidos, Java 8, em produção desde 2018.

Su2. Tipos de demanda esperados *
     (textarea, mínimo 10 caracteres)
     Ex: Bugs (40%), pequenas melhorias (40%), regulatório (20%).

Su3. SLA / criticidade do sistema *
     (seleção):
       - Baixa (resolução em dias)
       - Média (resolução em horas úteis)
       - Alta (24/7, resposta < 1h)


▌TIPO 9 — Outro (other)

Não possui perguntas type-specific. Apresenta APENAS as duas perguntas
comuns (C1 e C2). A especificação livre digitada no passo 1 (campo 4a —
"Descreva o tipo") é usada pelo Arguidor para escolher a stack.


───────────────────────────────────────────────────────────────────────────
RESUMO POR TIPO (quantidade total de perguntas exibidas)
───────────────────────────────────────────────────────────────────────────

  Passo 1 (básico):                5 campos (+1 se tipo = "Outro") = 5 ou 6
  Passo 2, por tipo (específicas + 2 comuns):
    new_system      → 4 + 2 = 6
    mobile_app      → 4 + 2 = 6
    module          → 3 + 2 = 5
    enhancement     → 3 + 2 = 5
    integration     → 4 + 2 = 6
    modernization   → 4 + 2 = 6
    etl             → 4 + 2 = 6
    maintenance     → 3 + 2 = 5
    other           → 0 + 2 = 2

  TOTAL (mínimo, tipo "other"): 5 + 2  = 7 perguntas
  TOTAL (máximo, qualquer tipo com 4 específicas + "Outro"): 6 + 6 = 12 perguntas
